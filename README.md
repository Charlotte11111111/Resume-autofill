## 1. 动机与痛点

### 1.1 问题背景

在求职过程中，海投是常态。一个求职者往往需要在多个招聘平台（BOSS直聘、智联招聘、猎聘、领英、企业官网）填写大量相似的个人信息。每次申请新职位，都需要：

- 重复填写姓名、邮箱、手机号
- 复制粘贴工作经历（通常 2-3 段）
- 填写教育背景（学校、专业、学历、时间）
- 填写项目经验、技能证书等

### 1.2 痛点分析

| 痛点 | 频率 | 影响 |
|------|------|------|
| 同一信息反复填写 | 每次申请 | 时间消耗大，效率低 |
| 复制粘贴容易出错 | 高 | 格式混乱，信息遗漏 |
| 不同平台字段名称不同 | 高 | 需要花费时间理解字段含义 |
| 网络不稳定导致表单超时 | 中 | 填写内容丢失，需要重来 |
| 简历 PDF 内容无法直接复用 | 高 | 需要手动对照填写 |

### 1.3 用户场景

**场景 A：海投求职者**
> 李明是一名软件工程师，在求职旺季每天投递 20-30 个职位。每个职位平均需要 10-15 分钟填写表单，其中 80% 的信息是重复的。

**场景 B：职场转型者**
> 张华从传统行业转向互联网，需要针对不同岗位调整简历内容。在填写时经常需要对比原简历和岗位要求，频繁切换窗口。

**场景 C：海归求职者**
> 王芳在国外留学多年，中文输入不熟练，填写中文表单时效率更低，而且很多专业术语需要查询。

## 2. 解决方案

### 2.1 核心思路

**一次解析，多次复用**

将简历（PDF/DOCX）解析为结构化数据存储，后续在任何招聘平台填写时，AI 自动识别字段并匹配填写。

### 2.2 产品定位

- **工具型产品**：简单、高效、专注解决一个问题
- **本地优先**：简历数据存储在本地，不上传服务器
- **AI 增强**：利用 AI 的语义理解能力处理复杂的字段匹配

### 2.3 核心价值

| 价值点 | 具体表现 |
|--------|----------|
| 节省时间 | 从每次 10-15 分钟缩短到 2-3 分钟 |
| 减少错误 | 避免复制粘贴导致的格式问题 |
| 保护隐私 | 简历数据本地存储，不经过第三方 |
| 跨平台统一 | 适配多种招聘网站，统一体验 |

## 3. 核心功能

### 3.1 简历解析

**功能描述**：上传简历文件（PDF/DOCX），AI 自动解析为结构化 JSON 数据。

**技术实现**：
- PDF 解析：使用 pdf.js 提取文本
- DOCX 解析：使用 mammoth.js 提取文本
- 结构化解析：调用 AI 接口，按照预设 Schema 提取信息

**解析字段**：
```
├── basicInfo (基础信息)
│   ├── fullName (姓名)
│   ├── email (邮箱)
│   ├── phone (电话)
│   ├── city (城市)
│   ├── country (国家)
│   ├── linkedIn (领英)
│   ├── website (个人网站)
│   └── github
├── education[] (教育经历，支持多段)
│   ├── school (学校)
│   ├── degree (学历)
│   ├── major (专业)
│   ├── startDate / endDate
│   └── description
├── workExperience[] (工作经历，支持多段)
│   ├── company (公司)
│   ├── title (职位)
│   ├── location (地点)
│   ├── startDate / endDate
│   └── description
├── projects[] (项目经历，支持多段)
│   ├── name
│   ├── role
│   ├── description
│   └── technologies
├── skills[] (技能列表)
├── languages[] (语言能力)
├── certifications[] (证书)
└── customFields (自定义字段，AI 学习补充)
```

### 3.2 智能匹配

**功能描述**：在招聘页面识别表单字段，AI 自动匹配简历数据。

**技术实现**：
1. **本地快速匹配**：基于字段名称关键词（label/name/id/placeholder）进行规则匹配
   - 姓名 → basicInfo.fullName
   - 邮箱 → basicInfo.email
   - 公司 → workExperience[0].company
   - ...

2. **AI 智能匹配**：本地无法识别的字段，交给 AI 分析语义后匹配
   - 示例："工作职责" → workExperience[0].description
   - 示例："项目名称" → projects[0].name

3. **分段隔离**：工作经历、教育经历等数组字段，按段落顺序隔离匹配
   - 第 N 个"公司名称"字段 → workExperience[N].company

**性能优化**：
- 首次匹配后缓存页面结构
- 同站点、同结构页面复用缓存
- 本地匹配优先，AI 仅处理未决字段

### 3.3 自动填写

**功能描述**：将匹配结果自动填入表单字段。

**填写规则**：
- 仅填写空字段，避免覆盖已有内容
- 支持 input、textarea、select 原生控件
- 对于多选项（如技能标签），智能拆分为多个选项

### 3.4 自定义字段学习

**功能描述**：手动填写的字段自动学习，下次复用。

**使用场景**：
- 某些平台的特殊字段 AI 无法识别
- 用户手动填写后，插件记住该字段映射
- 后续在同类型页面自动复用

## 8. 常见问题

### 8.1 安装问题

**Q: 扩展无法加载，显示"清单文件无效"**
A: 请确保选择的是项目根目录，而非某个子文件夹。

**Q: 弹出窗口空白**
A: 尝试刷新扩展（chrome://extensions/ → 刷新按钮），然后重新打开。

### 8.2 API 配置问题

**Q: API 调用失败 (401/403)**
A: 检查 API Key 是否正确，是否已过期。

**Q: API 调用失败 (404)**
A: 检查 Base URL 是否正确。正确格式：`https://api.minimax.chat`，错误格式：`https://api.minimax.chat/v1/text/chatcompletion_v2`（路径会自动拼接）。

**Q: API 调用成功但返回为空**
A: 可能是模型名称错误，请确认使用的模型名称与 AI 提供商后台显示一致。

### 8.3 简历解析问题

**Q: 解析结果为空或只有部分字段**
A: 可能原因：
1. PDF 是扫描件（图片格式），无法提取文字
2. PDF 加密保护
3. 简历格式非标准（如图形简历）

**Q: 教育经历/工作经历解析不完整**
A: 检查简历中是否有多段经历，确保每段有明确的时间分隔。

### 8.4 自动填写问题

**Q: 点击"在当前页面打开"没有反应**
A: 确保当前页面是普通 http/https 页面，而非 chrome:// 内部页面。

**Q: 某些字段没有被填写**
A: 这些字段可能是：
1. 非标准命名，AI 无法识别
2. 自定义下拉组件
3. 动态加载的字段
可以手动填写，插件会学习这个字段。

**Q: 填写后内容丢失**
A: 部分网站有防重复机制，填写后可能需要手动点击保存。

## 10. 安装说明

### 10.1 安装步骤

1. 打开 Chrome，访问 `chrome://extensions/`
2. 右上角开启「开发者模式」
3. 点击「加载已解压的扩展程序」
4. 选择本项目文件夹
5. 安装完成，工具栏出现插件图标

### 10.2 获取 MiniMax API Key

1. 访问 [MiniMax 开放平台](https://platform.minimaxi.com)
2. 注册/登录账号
3. 进入「API Keys」页面
4. 创建新的 API Key
5. 复制并填入插件设置

